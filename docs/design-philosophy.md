# 设计哲学

## 核心理念

OpenMicroChat 的设计不是追求功能堆砌，而是探索一种"用起来舒服"的聊天体验。以下原则贯穿整个项目的决策过程。

## 1. 触控优先，自然过渡

> 不常用功能通过特殊步骤自然切换过去，而不是右键菜单，从而兼容触控。

传统桌面应用依赖右键菜单承载次级操作，这在触控设备上完全失效。OpenMicroChat 采用手势驱动的方式暴露功能层级：

| 操作 | 触发方式 | 说明 |
|------|---------|------|
| 选择联系人 | 单击/点按 | 主路径，零学习成本 |
| 管理标签 | 长按联系人 | 自然的"深入"隐喻 |
| 进入设置 | 点击聊天头部 | 信息就在那里，点它就能看详情 |
| 切换主题 | 用户菜单 | 低频操作，藏在二级入口 |

关键点：**每一层交互都不需要说明书**。长按时的轻微缩放动画（scale 0.97）提供了"我正在进入深层操作"的触觉暗示。

## 2. 标签即视图

传统聊天应用的联系人分组是"文件夹"模型——一个人只能在一个文件夹里。OpenMicroChat 采用**标签模型**：

- 一个联系人可以同时属于"同事"和"朋友"
- 标签不是隐藏在搜索栏里的过滤条件，而是**可见的、可触摸的胶囊按钮**
- 选择标签后，韦恩图动画让"交集"这个抽象概念变得直觉可感

### 为什么是韦恩图

选择两个标签时，用户实际想问的是："谁同时是我的同事和朋友？" 韦恩图是这个问题最自然的可视化。三个区域（仅A、交集、仅B）可以点击，进一步缩小范围——这让集合运算变成了一种空间导航。

### 降级策略

三个及以上标签时韦恩图会变得混乱，此时退化为简单的胶囊排列 + 匹配计数。**不追求一致性的视觉形式，追求一致性的认知负担。**

## 3. 聚类 ≠ 分组

早期设计中有过"分组"的概念，后来明确了区分：

- **分组**：互斥的容器，一个人只能在一个组
- **聚类/标签**：重叠的属性，一个人可以有多个标签

这不只是数据模型的区别，更是**心理模型的区别**。用户脑中对"张三"的认知本身就是多维的（他既是同事、也是朋友、还是邻居），标签系统尊重这种认知的天然结构。

## 4. 登录前不应有联系人

早期版本在未登录状态下就展示了预设联系人列表，这带来两个问题：

1. **虚假承诺**：用户看到联系人会以为可以直接聊天，点击后发现需要登录，挫败感很强
2. **认知混乱**：这些是谁？是真实的还是假的？

解决方案：**未登录状态下侧边栏为空白**，登录后才加载用户自己的数据。体验模式提供预设数据，但通过"体验模式"的入口标签明确了这是演示。

## 5. 简洁是减法的结果

项目经历了多轮简化：

```
传统聊天应用全功能界面
    ↓ 参考现代移动端设计语言大幅简化
多场景设置页面系统
    ↓ 去掉独立的编辑模式
    ↓ 去掉独立的联系人列表页
    ↓ 聚类功能并入标签系统
当前形态：聊天列表 + 标签筛选 + 韦恩图
```

每一次减法都不是"删功能"，而是**找到更少的元素承载同样的能力**。标签胶囊栏取代了完整的编辑模式和联系人列表页，用更少的 UI 元素完成了更多的事。

## 6. 操作语义统一

同一个操作目的，无论从哪条路径触发，必须产生一致的结果。

具体实践：设置页面不是写了三套 HTML 的独立页面，而是一个动态容器，根据上下文（当前选中的是个人联系人还是群聊）渲染对应内容。无论用户从聊天头部点击还是从更多按钮进入，看到的内容和行为完全一致。

违反这个原则的典型场景：同一个"更新状态"的功能，从不同入口调用时携带了不同的上下文数据，导致结果不一致。

## 7. 防御性布局

**布局的首要责任是在任何显示条件下保持结构完整性，而不是只在理想条件下好看。**

具体实践：
- 所有 flex 容器的可伸缩子元素都加了 `min-height: 0` / `min-width: 0`，防止内容溢出
- 文本内容都有截断保护（`text-overflow: ellipsis`）
- 150% 缩放、窄窗口、移动端都不会出现内容被遮挡或按钮消失的情况
- 测试方法：如果可用空间减半，布局会怎样？——不能"变丑"到"功能损坏"的程度
