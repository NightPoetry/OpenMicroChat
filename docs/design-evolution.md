# 设计演进记录

本文档记录 OpenMicroChat 从概念到当前形态的设计决策过程，保留了关键的转折点和背后的思考。

## 阶段一：原型期

### 起点
最初目标是搭建一个标准的即时通讯界面原型，包含联系人列表、聊天窗口、消息输入等基本要素。

### 第一次方向调整：视觉简化
原型完成后，发现界面元素过多、视觉层次不够清晰。决定参考现代移动端的设计语言进行大幅简化：

- 去掉多余的装饰元素
- 线条分隔替代色块分区（`#E5E5E5` 细线）
- 统一圆角、间距、字号体系
- 简洁、克制、留白

## 阶段二：设置系统设计

### 多场景设置页面
设计了完整的设置页面系统，定义了清晰的入口规范：

| 页面 | 入口方式 |
|------|---------|
| 用户设置 | 点击侧边栏用户头像 |
| 好友设置 | 聊天窗口右上角设置按钮 |
| 群聊设置 | 群聊窗口顶部区域 |
| 群聊好友设置 | 群聊成员列表中长按好友头像 |

群聊好友设置的入口设计值得说明：长按头像触发 1.5 倍放大 + 0.3 秒渐隐过渡动画后进入。这是为了让触控操作有明确的视觉反馈，同时暗示"正在进入更深层的信息"。

## 阶段三：聚类/标签系统探索

### 编辑模式的引入与废弃

最初为联系人管理设计了"编辑模式"：

1. **长按联系人头像** → 进入编辑模式
2. 编辑模式下联系人前出现**复选框**，支持多选
3. 拖动多个联系人到聚类区域完成分组
4. 进入瞬间有一次轻微抖动反馈（避免持续晃动）

关键设计决策记录：

- **"不需要联系人列表，所有联系人一定在消息记录里"** → 去掉了独立的联系人列表 tab，消息列表本身就是联系人列表
- **"那叫聚类，不叫分组。是打标签，只是视觉上看起来像分组"** → 确立了标签而非互斥分组的心理模型
- **"标签系统要自然，看起来像文件夹，而不是通过搜索来找"** → 标签应该是可见的、空间化的，不是隐藏在搜索背后的过滤器

### 编辑模式的简化过程

编辑模式经历了多次迭代：

```
v1: 编辑模式 + 联系人列表 + 聚类面板（三栏）
    ↓ 太复杂
v2: 编辑模式 + 标签分组/文件夹分组双 tab
    ↓ tab 切换增加了认知成本
v3: 简化为纯标签分组
    ↓ 独立的编辑模式仍然太重
v4: 去掉编辑模式，标签管理通过长按联系人触发
```

最终决策：**编辑模式作为一个独立的"模态"太重了**。标签管理应该是就地完成的轻量操作——长按联系人，弹出标签选择面板，点选完毕自动关闭。

### 韦恩图的引入

标签筛选的可视化经历了从列表到图形的演变：

- 最初设想：搜索栏输入标签名 → 过滤联系人列表
- 改进：标签作为可点击的胶囊按钮排列在搜索栏下方
- 最终：选中标签后用韦恩图动画展示集合关系
  - 一个标签 → 一个圆，弹性缩放动画
  - 两个标签 → 两个圆重叠，展示交集区域和各自独有区域
  - 三个及以上 → 降级为胶囊 + 计数（韦恩图三圆以上变得难以理解）

## 阶段四：后端集成

### 从纯前端到全栈

项目最初是纯前端原型，后来明确了需要真正可用的后端能力：

- JWT 认证系统（注册/登录）
- 单文件数据库（lowdb）替代 MongoDB，降低部署门槛
- 文件传输、语音录制等多媒体能力
- Socket.io 实时通信

### 关键经验："不能因为有了后端就让前端效果变形"

在集成后端的过程中出现过一个问题：后端功能的加入导致前端交互逻辑被破坏（例如登录后直接弹出了不应出现的编辑页面）。这引出了一个原则：**后端是为前端体验服务的，不能反过来让前端迁就后端的数据结构。**

### 体验模式的加入

为了解决"用户还没注册就想看看应用长什么样"的需求，加入了体验模式：
- 预设联系人和消息数据
- 消息发送有模拟回复
- 无需注册，点击即用
- 明确标记为"体验模式"，不会造成认知混淆

## 阶段五：当前架构

### 最终技术选型

| 层次 | 选择 | 原因 |
|------|------|------|
| 前端框架 | 无（原生 ES Modules） | 项目规模适中，避免框架带来的复杂度 |
| 状态管理 | 自建发布-订阅 Store | 足够轻量，无需 Redux 级别的工具 |
| 样式方案 | CSS 自定义属性 | 原生支持主题切换，无构建步骤 |
| 后端 | Express + lowdb | 单文件部署，零配置数据库 |
| 实时通信 | Socket.io | 成熟方案，浏览器兼容性好 |
| 认证 | JWT | 无状态，前后端分离友好 |
